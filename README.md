# Tutorial: pageoracle 
PageOracle — это настольное приложение на базе искусственного интеллекта, разработанное для того, чтобы помочь пользователям взаимодействовать с учебниками. Оно позволяет загружать книги (например, .txt файлы), задавать вопросы об их содержании и получать интеллектуальные ответы, сгенерированные искусственным интеллектом. Система преобразует книги в базу знаний с возможностью поиска, интегрируется с различными большими языковыми моделями и управляет историей ваших диалогов с помощью интуитивно понятного ИИ-агента на основе графов и удобного графического интерфейса.

## Пользовательский интерфейс (GUI)

1.	Что такое GUI: Это визуальная «панель управления» программой, заменяющая сложные текстовые команды на кнопки и окна. В PageOracle он реализован на языке Python с помощью библиотеки Tkinter.
2. Основные элементы:
• Боковая панель: кнопки загрузки книг и настроек.
• Область чата: история диалога с ИИ и поле ввода вопросов.
• Системные журналы: панель внизу для отслеживания фоновых процессов (логов).
3. Как пользоваться:
• Запустить приложение командой python main.py.
• Загрузить текстовый файл (.txt) через кнопку «Load Book».
• Задать вопрос в поле ввода и получить ответ от ИИ.
4. Техническая часть: GUI выступает посредником. Когда пользователь нажимает кнопки, интерфейс передает запросы на бэкенд (логическую часть), где происходит основная обработка данных. В коде за структуру окна отвечает класс PageOracleApp.

## Обработка заказов и база знаний

1. Зачем это нужно: ИИ не может «проглотить» огромную книгу целиком из-за ограничений на объем памяти (токенов). Поэтому книгу нужно предварительно подготовить.
2. Разбиение (Фрагментация): Текст делится на небольшие отрезки (чанки) по 1000 символов с перекрытием в 200 символов. Это позволяет сохранить смысл и контекст в каждом кусочке.
3. Аннотирование (Метаданные): К каждому фрагменту добавляется информация о том, из какой книги, части или главы он взят. Для поиска заголовков в коде используются регулярные выражения.
4. Создание эмбеддингов: Текст преобразуется в математические векторы (числовые «отпечатки») с помощью модели BAAI/bge-m3. Это позволяет ИИ искать информацию не по точным словам, а по смыслу.
5. Хранение в векторной базе: Все данные сохраняются в базу данных Chroma. Она работает как «умный библиотекарь», мгновенно находя нужные фрагменты текста при поступлении вопроса.

## Настройки приложения

1. Назначение настроек: Это «панель управления», которая позволяет менять «топливо» (поставщика ИИ) и регулировать «скорость и маневренность» (параметры генерации).
2. Основные категории настроек:
• Поставщик / Модель: выбор между OpenAI, DeepSeek и другими.
• Ключ API: секретный код для доступа к мощностям выбранного ИИ.
• Параметры генерации: Temperature (уровень креативности), Max Tokens (длина ответа).
• Параметры поиска: Score Threshold (фильтр релевантности информации из книг).
3. Техническая реализация:
• Все настройки сохраняются в файл settings.json на компьютере пользователя.
• Приложение использует функции load_settings() и save_settings() для работы с этим файлом.
• Для динамического переключения между моделями без перезагрузки кода используется библиотека importlib.
4. Процесс применения: После нажатия кнопки «Сохранить» приложение обновляет конфигурацию на бэкенде, пересоздает цепочки логики и выводит подтверждение в системный журнал.

## Интеграция и настройка больших языковых моделей

1. Что такое LLM: Это мощные системы ИИ (как GPT-4 или DeepSeek), которые понимают текст и генерируют ответы. Интеграция позволяет приложению переключаться между ними, как между разными двигателями в автомобиле.
2. Ключевые концепции:
• Поставщики (Providers): компании, предоставляющие доступ к моделям.
• Ключ API: секретный пароль для авторизации и оплаты услуг ИИ.
• Динамическая загрузка: технология, позволяющая программе загружать нужную библиотеку ИИ «на лету» только тогда, когда она выбрана в настройках.
3. Параметры генерации (регуляторы):
• Temperature: уровень креативности (низкая — для фактов, высокая — для фантазии).
• Max Tokens: ограничение длины ответа.
• Top P: разнообразие используемых слов.
4. Техническая реализация:
• В коде используется словарь PROVIDERS, где прописаны пути к классам и пакетам для каждого поставщика (например, langchain_openai).
• Основная функция set_model динамически импортирует нужные модули с помощью importlib, настраивает API-ключ в переменных среды и создает объект модели с заданными параметрами.
• После смены модели приложение автоматически перестраивает свои внутренние логические цепочки.

## Механизмы поиска

1. Зачем нужен поиск: Даже самый мощный ИИ не может помнить содержание тысяч страниц. Механизмы поиска служат связующим звеном, которое находит в базе Chroma только те фрагменты, которые реально нужны для ответа на конкретный вопрос.
2. Как это работает (Эмбеддинги): Вопрос пользователя превращается в числовой вектор (отпечаток). Затем поисковая система сравнивает его с отпечатками фрагментов книг и выбирает наиболее похожие по смыслу.
3. Стратегии поиска (Ретриверы):
• Поиск сходства (Similarity Search): находит максимально близкие по значению фрагменты. Идеально для поиска конкретных цитат.
• MMR (Maximal Marginal Relevance): балансирует между релевантностью и разнообразием. Эта стратегия следит, чтобы ИИ не получил 10 одинаковых по смыслу абзацев, а видел разные аспекты темы.
4. Фильтрация (Score Threshold): Приложение использует настраиваемый «порог качества». Если фрагмент лишь отдаленно напоминает тему вопроса, он отсеивается и не передается ИИ, чтобы не вводить его в заблуждение.
5. Путь вопроса:
• Вопрос → Вектор → Поиск в Chroma → Извлечение фрагментов с метаданными → Передача их в ИИ как контекста.

## Диалоговый ИИ-агент (LangGraph)

1. Суть агента: Это «мозг» системы, который не просто выдает ответы, а ведет осмысленный диалог: понимает намерения, принимает решения (где искать данные) и управляет ходом беседы.
2. Как устроен LangGraph:
• Граф: общая структура (блок-схема).
• Узлы (Nodes): конкретные шаги (анализ вопроса, поиск в книгах, генерация ответа).
• Состояние (State): «записная книжка» агента, где хранится вся информация о текущем диалоге.
3. Процесс работы в режиме «Авто»:
• Анализ: Бот определяет, о чем вопрос (о приложении или по содержанию книг).
• Поиск: Ищет информацию в базе знаний (книгах).
• Маршрутизация: Если данных мало — просит уточнить вопрос; если нашел — отвечает по тексту; если вопрос общий — отвечает на основе знаний нейросети.
• Генерация: Формирует итоговый ответ пользователю.

## История чата и управление контекстом

1. Зачем это нужно: История чата позволяет ИИ понимать контекст диалога. Благодаря ей можно задавать уточняющие вопросы (например, «Кто его автор?», где «его» — это книга из предыдущего сообщения), не повторяя детали каждый раз.
2. Два уровня памяти:
• Кратковременная: активный список сообщений в текущей сессии.
• Долговременная: сохранение диалогов в файл chat_history.json, что позволяет продолжать общение даже после перезапуска приложения.
3. Управление контекстом (Обрезка): Чтобы ИИ не путался в слишком длинных диалогах и не тратил лишние ресурсы, система хранит только последние 20 сообщений (метод _trim_history).
4. Функции в интерфейсе:
• Очистить чат: просто скрывает сообщения с экрана, но ИИ продолжает их помнить.
• Очистить историю: полностью удаляет файл памяти, и ИИ «забывает» всё общение.
5. Техническая реализация: Сообщения хранятся в виде списка словарей. Перед отправкой в нейросеть они преобразуются в специальные объекты (HumanMessage и AIMessage), которые подставляются в шаблон промпта.
